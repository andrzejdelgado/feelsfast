# View Transitions API

The View Transitions API is the web platform's answer to a problem framework authors have been re-solving for fifteen years: smooth animated transitions between two states of a page, without requiring a single component tree to own both states. The browser snapshots the old state, lets the new state render, then animates between the two snapshots automatically.

It lands in the perceived-performance catalog because the alternatives — manual orchestration with FLIP, route-aware AnimatePresence wrappers, scroll-position juggling — are expensive to maintain and easy to break. The API does the work for you, with timings that match Card et al. 1991's <Cite id="card-1991" label="Card et al. 1991" /> ~10 Hz refresh budget and Card-Moran-Newell's <Cite id="card-moran-newell-1983" label="Card, Moran & Newell 1983" /> ~100 ms perceptual frame as the lower bound for "this feels like motion."

Modern browser support is broad enough as of 2025 to use it as the primary route-transition mechanism, with a graceful fallback to instant swap for the unsupported set. Next.js, Astro, SvelteKit, and Remix all expose it through their navigation primitives; the manual API is small enough to use directly.

## When to use

1. **Route transitions** — the headline use case. Cross-document support means even multi-page apps get the smooth handoff.
2. **Same-document state changes** with a stable visual relationship between the old and new states (e.g. a card expanding into a detail view).
3. **Modal opens / closes** where the trigger and the modal share visual identity.

## When *not* to use

1. **Below 100 ms** — Card-Moran-Newell's perceptual frame. A transition shorter than that registers as a discrete jump anyway.
2. **Where the old and new states are visually unrelated.** A transition between two completely different pages just adds 200 ms of motion the user has to wait through.
3. **For users with `prefers-reduced-motion`** — substitute a fade or skip the animation entirely.

## Code

```ts
// Manual same-document transition.
function navigate(updateDOM: () => void) {
  if (!document.startViewTransition) {
    // Browser without support — apply the change directly.
    updateDOM();
    return;
  }

  const transition = document.startViewTransition(() => {
    updateDOM();
  });

  // Optional: tune the animation per-element via CSS, or hook into
  // transition.finished / .ready / .updateCallbackDone for sequencing.
}
```

```css
/* Default cross-fade between snapshots. */
::view-transition-old(root),
::view-transition-new(root) {
  animation-duration: 200ms;
  animation-timing-function: cubic-bezier(0, 0, 0.2, 1);
}

/* Per-named-element morphs. The two elements share a `view-transition-name`
   set in component CSS (e.g. on a card and the same item's detail view). */
::view-transition-old(card-1),
::view-transition-new(card-1) {
  animation-duration: 250ms;
}

/* Reduced-motion override — skip the animation entirely. */
@media (prefers-reduced-motion: reduce) {
  ::view-transition-old(root),
  ::view-transition-new(root) {
    animation-duration: 0s;
  }
}
```

For Next.js App Router, the framework provides `unstable_ViewTransition` for opt-in cross-route transitions; consult the bundled docs at `node_modules/next/dist/docs/01-app/02-guides/instant-navigation.mdx` (or whichever section your version exposes) before wiring it up — the API has changed across recent releases.

## Accessibility

1. **`prefers-reduced-motion` override** is mandatory. The API does not respect the system preference automatically; you must short-circuit `animation-duration` to zero, or skip the transition call entirely.
2. **Focus management** — when the transition crosses route or content boundaries, ensure focus moves to the relevant landmark in the new state.
3. **Don't trap announcements** — assistive tech ignores the transition itself, but if you delay rendering the new content's heading or main landmark, the announcement comes late.
4. **Keep durations under ~300 ms** for state changes, ~400 ms for route transitions. Above that, the animation becomes a wait rather than a transition.
