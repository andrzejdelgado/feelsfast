# Touch start / pointer down on touch

<RelatedRail of="touch-pointer-down" kind="pattern" />

The folkloric "100 ms feels instant" rule is right for content updates and form submissions. It is wrong by an order of magnitude for direct manipulation. Jota et al. 2013 <Cite id="jota-2013" label="Jota et al. 2013" /> measured touch-input JNDs and found users could discriminate latencies as low as ~33 ms for dragging — roughly a third of the 100 ms threshold for content updates. Tapping was higher (~82 ms) but still well below 100 ms. Deber et al. 2015 <Cite id="deber-2015" label="Deber et al. 2015" /> confirmed and extended these numbers; Ng et al. 2012 <Cite id="ng-2012" label="Ng et al. 2012" /> earlier showed experienced users detecting improvements down to single-millisecond ranges.

The practical consequence: if your slider, drag, or scroll has 100 ms of latency, it does not "feel instant." It feels noticeably laggy, and there is no skeleton screen or progress bar that fixes it. The user is waiting for the cursor to catch up to their finger. The fix is real engineering: hardware-accelerated transforms, reduced JS work on the main thread, and listening on `touchstart` / `pointerdown` with appropriate gating.

This pattern complements [mousedown / pointerdown vs. click](/patterns/mousedown-vs-click) by handling the touch-specific concerns: the `touchmove` threshold (so scrolling doesn't fire actions), the passive-listener implications, the platform-specific quirks.

## When to use

1. **Direct-manipulation surfaces** — sliders, drag handles, scrubbers, swipe gestures, draggable list items.
2. **Touch-friendly buttons** that need the same head-start the [mousedown vs. click](/patterns/mousedown-vs-click) pattern delivers on desktop.
3. **Custom gestures** where the visible motion needs to track the finger in real time.

## When *not* to use

1. **Any interaction that should respect scroll.** A button inside a scrollable list will fire on `touchstart` whenever the user's finger crosses it during a scroll — needs a `touchmove` gate.
2. **Side-effects with cost or commitment** — same as the mouse case; `touchstart` should only do reversible, idempotent work.
3. **Without measuring the actual end-to-end latency.** Listener choice is one part of the story; render performance is the other.

## Code

```tsx
"use client";

import { useRef, type PointerEvent } from "react";

const MOVE_THRESHOLD_PX = 6;

export function TapWithMoveGate({
  onTap,
  children,
}: {
  onTap: () => void;
  children: React.ReactNode;
}) {
  const start = useRef<{ x: number; y: number } | null>(null);
  const cancelled = useRef(false);

  return (
    <button
      type="button"
      // Use Pointer Events for unified mouse/touch/pen handling.
      onPointerDown={(event: PointerEvent<HTMLButtonElement>) => {
        start.current = { x: event.clientX, y: event.clientY };
        cancelled.current = false;
      }}
      onPointerMove={(event: PointerEvent<HTMLButtonElement>) => {
        if (!start.current || cancelled.current) return;
        const dx = event.clientX - start.current.x;
        const dy = event.clientY - start.current.y;
        if (Math.hypot(dx, dy) > MOVE_THRESHOLD_PX) {
          cancelled.current = true;
        }
      }}
      onPointerUp={() => {
        if (!cancelled.current) onTap();
        start.current = null;
        cancelled.current = false;
      }}
    >
      {children}
    </button>
  );
}
```

For drag interactions, the latency budget is tighter. Use `touch-action: none` on the dragged element to short-circuit the browser's default scroll handling, and run the position update inside `requestAnimationFrame`:

```tsx
"use client";

import { useEffect, useRef, useState } from "react";

export function Slider() {
  const [value, setValue] = useState(50);
  const dragging = useRef(false);
  const trackRef = useRef<HTMLDivElement>(null);

  function commit(clientX: number) {
    const track = trackRef.current;
    if (!track) return;
    const rect = track.getBoundingClientRect();
    const ratio = (clientX - rect.left) / rect.width;
    setValue(Math.max(0, Math.min(100, ratio * 100)));
  }

  function onPointerMove(event: PointerEvent) {
    if (!dragging.current) return;
    requestAnimationFrame(() => commit(event.clientX));
  }

  useEffect(() => {
    function onUp() {
      dragging.current = false;
    }
    window.addEventListener("pointermove", onPointerMove);
    window.addEventListener("pointerup", onUp);
    return () => {
      window.removeEventListener("pointermove", onPointerMove);
      window.removeEventListener("pointerup", onUp);
    };
  }, []);

  return (
    <div
      ref={trackRef}
      style={{ touchAction: "none" }}
      className="relative h-2 w-full rounded-full bg-muted"
      onPointerDown={(event) => {
        dragging.current = true;
        commit(event.clientX);
      }}
    >
      <div
        className="absolute top-1/2 size-4 -translate-y-1/2 rounded-full bg-primary"
        style={{ left: `calc(${value}% - 8px)` }}
      />
    </div>
  );
}
```

`touch-action: none` is what makes this *feel* responsive; without it, mobile browsers wait ~300 ms for gesture disambiguation before firing the events.

## Accessibility

1. **Don't lose keyboard support.** The pointer-event handlers are for pointer/touch; `Enter` and `Space` should still trigger the button via the standard `onClick`. Actual sliders should respond to arrow keys.
2. **`role="slider"`, `aria-valuenow / valuemin / valuemax`** on custom slider implementations — the visual is half the work; the ARIA contract is the other half.
3. **Sufficient hit-target size.** Touch targets should be ≥44 × 44 px (Apple HIG) or ≥48 × 48 px (Material) — direct-manipulation precision matters even more under low latency.
4. **`prefers-reduced-motion`** does not apply directly here — input responsiveness is not animation — but any *visual* feedback you tie to the pointer (a highlight, a press state) should respect it.
