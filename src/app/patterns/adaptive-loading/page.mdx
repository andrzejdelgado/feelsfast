# Adaptive loading (performance scaler)

Most teams calibrate their loading patterns to one slice of their user distribution — typically the dev-team's own machines on a fast office network. The skeletons shimmer for 800 ms because that is the median wait the team experienced during development. The user on a 3G connection from a moving train sees the same 800 ms skeleton over a wait that actually lasts 6 s, and the affordance bears no relationship to their reality.

Eli Fitch <Cite id="fitch" label="Fitch" /> calls the fix the *performance scaler*. Measure each user's actual round-trip times. Compute the ratio between observed and expected. Apply that ratio to every loading affordance the user encounters. The same code path produces appropriate perception treatment for the user's actual conditions — fast connections see no spinner where slow connections see a determinate progress bar; mid-range connections see a skeleton calibrated to their typical wait.

This is not a perception trick. It is honesty engineering. A fixed 800 ms skeleton is a guess at what most users experience; a scaler-driven skeleton matches each user's reality. Every other perception pattern in your product (the skeleton, the spinner threshold, the engagement boundary) reads correctly across the distribution instead of being calibrated to one slice.

The pattern aligns with Doherty 1982's <Cite id="doherty-1982" label="Doherty 1982" /> finding that response-time productivity is a continuous curve, not a fixed threshold. Adaptive loading treats each user's position on that curve as the input, and matches the perception layer to it.

## When to use

1. **Apps with a wide user distribution** — geography, device class, network class. The wider the distribution, the bigger the gain.
2. **Production-scale products** where you can collect real-user telemetry. The scaler needs data.
3. **Layouts with multiple loading affordances** that share the same calibration logic. Building the scaler once and applying it everywhere is the leverage move.

## When *not* to use

1. **Single-environment apps** (internal tools where everyone is on the same network). The variance is small; the scaler is overhead.
2. **Apps with very short waits everywhere.** If nothing in your product exceeds 1 s for any user, adaptive loading is moot.
3. **Without normalising for resource size or call complexity.** A scaler that mixes "fetch a small image" and "fetch a large image" produces nonsense; normalise per request type before scaling.

## Code

A minimal client-side performance scaler. Track recent request durations, compute a rolling ratio against expected baselines, expose it as a hook.

```tsx
// src/lib/perf-scaler.ts

const RECENT_WINDOW = 10; // requests
const recent: { observed: number; expected: number }[] = [];
let cachedScaler = 1;

export function recordRequest(observedMs: number, expectedMs: number) {
  if (expectedMs <= 0) return;
  recent.push({ observed: observedMs, expected: expectedMs });
  if (recent.length > RECENT_WINDOW) recent.shift();

  const totalObs = recent.reduce((s, r) => s + r.observed, 0);
  const totalExp = recent.reduce((s, r) => s + r.expected, 0);
  cachedScaler = totalExp === 0 ? 1 : totalObs / totalExp;
}

export function getScaler(): number {
  return cachedScaler;
}
```

```tsx
"use client";

import { useEffect, useState } from "react";
import { getScaler } from "@/lib/perf-scaler";

export function useScaledDuration(baseMs: number): number {
  const [scaled, setScaled] = useState(baseMs);
  useEffect(() => {
    setScaled(baseMs * getScaler());
  }, [baseMs]);
  return scaled;
}
```

A skeleton component then takes its visible duration from the scaler:

```tsx
function SkeletonRow() {
  // Base duration calibrated to the team's average; scales per-user.
  const shimmerMs = useScaledDuration(1400);
  return (
    <div
      style={{ animationDuration: `${shimmerMs}ms` }}
      className="h-4 w-32 animate-pulse rounded bg-muted motion-reduce:animate-none"
    />
  );
}
```

For a more advanced version, normalise by request type so a small image fetch does not pollute the scaler with a different baseline than a large data query:

```tsx
const RECENT_PER_TYPE: Record<string, { observed: number; expected: number }[]> = {};

export function recordRequestTyped(
  type: string,
  observedMs: number,
  expectedMs: number,
) {
  if (!RECENT_PER_TYPE[type]) RECENT_PER_TYPE[type] = [];
  RECENT_PER_TYPE[type].push({ observed: observedMs, expected: expectedMs });
  if (RECENT_PER_TYPE[type].length > RECENT_WINDOW) {
    RECENT_PER_TYPE[type].shift();
  }
}
```

Production implementations also handle baseline drift, integration-test calibration, and weighting recent samples more heavily. The principle stays the same: measure reality, scale to match.

## Accessibility

1. **The scaler is invisible to users.** No accessibility implications directly — but every pattern it influences (skeleton duration, animation timing, spinner thresholds) carries its own a11y notes.
2. **`prefers-reduced-motion`** still applies. The scaler scales the *duration* of an animation; reduced-motion users skip the animation entirely.
3. **`prefers-reduced-data`** users implicitly need a more conservative scaler — even a fast-network experience should not auto-prefetch aggressively for them.
