# Token-by-token streaming render

An LLM produces its response as a sequence of tokens, not a single block. Streaming surfaces those tokens to the user as they arrive instead of buffering the whole response. The total time to last token is the same; the time to first content is dramatically shorter; the user reads along with the model rather than waiting for it to finish thinking.

The pattern earns its keep on two fronts. Prospectively, Block & Zakay 1997's <Cite id="block-zakay-1997" label="Block & Zakay 1997" /> filled-duration principle compresses how the wait registers — the user has something to encode, the wait feels filled. Card et al. 1991's <Cite id="card-1991" label="Card et al. 1991" /> ~100 ms perceptual frame sets the *floor* for the cadence — too fast and the eye cannot keep up with reveals, too slow and the wait feels mechanical.

The non-obvious detail: the cadence matters more than the latency. Instant per-token reveal jitters the eye and disrupts the encoding. A constant-rate metronomic reveal feels like a 1990s teleprinter. Variable chunk sizes (1–3 characters at a time, slight randomness) feel right — closer to natural reading speed than either extreme. The live demo on [/playground](/playground) shows the difference.

## When to use

1. **Any AI-generated text response longer than ~2 seconds total** — chat responses, code generation, document summarisation.
2. **Where the user can usefully read partial output** — coherent prose, structured lists, code that compiles incrementally.
3. **Pair with a [thinking state](/patterns/ai-thinking-state)** for the time-to-first-token gap.

## When *not* to use

1. **Outputs that are not meaningful in partial form** — a JSON blob the user will copy as a unit; a single number; a yes/no answer.
2. **Very fast responses** (under 500 ms total) where the streaming overhead and the variable-chunk pacing add visible jitter.
3. **For users with `prefers-reduced-motion`** — fall back to revealing complete sentences or paragraphs, not character-by-character.

## Code

A minimal token-streaming renderer. The example uses a generator that chunks an in-memory string; production implementations consume Server-Sent Events or a fetch stream from the model API.

```tsx
"use client";

import { useEffect, useRef, useState } from "react";

const TICK_MS = 30;

export function StreamedText({
  source,
  onDone,
}: {
  source: AsyncIterable<string>;
  onDone?: () => void;
}) {
  const [text, setText] = useState("");
  const [streaming, setStreaming] = useState(true);

  useEffect(() => {
    let cancelled = false;
    let buffer = "";
    let revealed = "";
    let done = false;

    async function consume() {
      for await (const chunk of source) {
        if (cancelled) return;
        buffer += chunk;
      }
      done = true;
    }

    function tick() {
      if (cancelled) return;
      // Variable chunk: 1–3 characters at a time, weighted toward 1.
      const r = Math.random();
      const chunkSize = r < 0.7 ? 1 : r < 0.95 ? 2 : 3;
      const next = buffer.slice(0, chunkSize);
      buffer = buffer.slice(chunkSize);
      if (next) {
        revealed += next;
        setText(revealed);
      }
      if (done && buffer.length === 0) {
        setStreaming(false);
        onDone?.();
        return;
      }
      setTimeout(tick, TICK_MS);
    }

    void consume();
    setTimeout(tick, TICK_MS);

    return () => {
      cancelled = true;
    };
  }, [source, onDone]);

  return (
    <span aria-live="polite" aria-busy={streaming}>
      {text}
      {streaming ? <Cursor /> : null}
    </span>
  );
}

function Cursor() {
  return (
    <span
      aria-hidden="true"
      className="ml-0.5 inline-block h-3.5 w-1 translate-y-0.5 bg-primary motion-safe:animate-pulse"
    />
  );
}
```

The split between the network-layer consumer (filling `buffer`) and the render-layer ticker (draining `buffer` into `revealed`) is the important structural choice. It decouples the cadence from the model's actual token rate — a slow model still gets paced reveals, a fast model does not jitter the eye with too-fast updates.

## Accessibility

1. **`aria-live="polite"`** on the streaming container so screen readers announce updates as they happen.
2. **`aria-busy="true"`** during streaming, flipping to `false` on completion.
3. **`prefers-reduced-motion`** — the cursor's pulsing animation should drop; consider also slowing the reveal cadence so it feels more like paragraph reveals than character ones.
4. **Don't strip semantics from the streamed output.** If the model is producing markdown, render it with the right elements as the tokens accumulate. A streaming `<pre><code>` block reads better than a paragraph that retroactively becomes a code block.
