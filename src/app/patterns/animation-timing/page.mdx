# Animation timing for state changes

<RelatedRail of="animation-timing" kind="pattern" />

Most UIs ship animations at one of two durations: too fast to read or too slow to wait for. The right answer is neither — the duration depends on the cognitive transition the animation is bridging. Three families cover the vast majority of state changes.

The durations are not arbitrary. Card et al. 1991 <Cite id="card-1991" label="Card et al. 1991" /> recommended ~10 Hz refresh during transitions to maintain object permanence — that is the ~100 ms-per-frame budget the families are derived from. Card-Moran-Newell 1983 <Cite id="card-moran-newell-1983" label="Card, Moran & Newell 1983" /> set the lower bound: below ~100 ms, an animation reads as a discrete jump, not as motion. Fitch <Cite id="fitch" label="Fitch" /> measured the 200 ms sweet spot for press-feedback animations specifically. Harrison et al. 2010 <Cite id="harrison-2010" label="Harrison et al. 2010" /> argued for deceleration over linear or acceleration as the default curve — bars that finish strong feel faster.

| Duration | Use for | Curve |
|---|---|---|
| **~100 ms** | Hover states, tooltips, focus rings, instant acknowledgments | `ease-out` |
| **~200 ms** | Button press, small state changes, list item entry/exit, active-state animations | `ease-out` |
| **~400 ms** | Modal opens, panel slides, route transitions, micro-celebrations | `cubic-bezier(0, 0, 0.2, 1)` |

Below ~100 ms, the change is too fast to register as motion; the user sees the new state without the bridge. Above ~400 ms, the animation is a wait, not a transition — past 500 ms users start to feel they are watching the UI work rather than respond to them.

## When to use

These durations are defaults, not rules. They cover ~90 % of cases. Custom durations are appropriate when:

1. The user's context demands a longer cue (a celebration animation marking a meaningful action).
2. The animation is part of an explicit narrative (an onboarding sequence with paced reveals).
3. The animation is *delaying* an action deliberately (a confirmation dialog that needs ~600 ms to register).

## When *not* to use

1. **Below 100 ms.** It does not register as motion; just snap to the new state.
2. **Above 500 ms for routine state changes.** A 1-second slide between tabs is not a transition; it is a short loading screen.
3. **In contexts where the user is already doing something else.** A 400 ms modal entry while the user is typing is friction; cut to 200 ms or skip the animation.

## Code

```css
/* Three families exposed as design tokens. */
@theme inline {
  --duration-quick: 100ms;
  --duration-base: 200ms;
  --duration-deliberate: 400ms;

  --ease-out: cubic-bezier(0, 0, 0.2, 1);
}

/* Hover, focus, and other ~100 ms transitions. */
.tooltip {
  transition:
    opacity var(--duration-quick) var(--ease-out),
    transform var(--duration-quick) var(--ease-out);
}

/* Button press, list item enter/exit, active-state animations. */
.button {
  transition: transform var(--duration-base) var(--ease-out);
}

.button:active {
  transform: scale(0.97);
}

/* Modal, panel, route transition. */
.modal {
  transition:
    opacity var(--duration-deliberate) var(--ease-out),
    transform var(--duration-deliberate) var(--ease-out);
}
```

```tsx
// In Tailwind v4 the same families surface as utility classes.
<button className="transition-transform duration-200 ease-out active:scale-[0.97]">
  Save
</button>

<div className="
  fixed inset-0 bg-black/50 transition-opacity duration-300 ease-out
  data-[state=open]:opacity-100 data-[state=closed]:opacity-0
">
  …
</div>
```

The exact duration depends on the perceived weight of the element. A small button at 200 ms feels right; a large card sliding in feels sluggish at 200 ms and right at 280–320 ms. The families are anchors, not constants.

## Accessibility

1. **`prefers-reduced-motion: reduce`** is mandatory. For each family, fall back to `transition: none` or to opacity-only transitions:

   ```css
   @media (prefers-reduced-motion: reduce) {
     .tooltip,
     .button,
     .modal {
       transition: none;
     }
   }
   ```

2. **Don't animate properties that cause reflow** (height, width, top, left). Stick to `transform` and `opacity` for compositor-only animations.

3. **Don't block input on the duration of the animation.** A 400 ms modal entrance that intercepts clicks for the full duration is a 400 ms input-latency hit.

4. **For animations that move focus** (modal open, panel reveal), move focus *before* or *after* the animation, never during. Mid-flight focus moves create chaotic screen-reader announcements.
