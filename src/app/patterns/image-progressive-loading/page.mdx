# Image progressive loading (LQIP, blur-up)

<RelatedRail of="image-progressive-loading" kind="pattern" />

A full image is heavy. A 4 KB placeholder embedded in the markup, blurred and stretched to the full dimensions, is light. Show the placeholder first; swap the real image in once it has loaded. The user sees roughly the right shape and colour within ~80 ms; the texture catches up over the next half-second. The wait is the same in clock terms; the experience is meaningfully fuller.

The mechanism cuts across two perception findings. James 1890 <Cite id="james-1890" label="James 1890" /> on filled duration is the broad reason: a wait with content the user can encode feels shorter than a wait with nothing. Anstis 2001 <Cite id="anstis-2001" label="Anstis 2001" /> and 2003 <Cite id="anstis-2003" label="Anstis 2003" /> on contrast and motion are the specific reason the *kind* of placeholder matters: edge contrast modulates how alive the image feels as it loads. A high-contrast LQIP — sharper edges, even if the textures are smudged — reads as faster than a soft, ghostly placeholder.

The two common variants:

1. **LQIP (Low-Quality Image Placeholder)** — a tiny base64-encoded JPEG/WebP at maybe 16×16 px, scaled to the final size. Cheap to generate, cheap to deliver, works in any framework.
2. **BlurHash / Plaiceholder / Vercel Image** — a compact string or hash that decodes into a colour-and-shape approximation. More compact than LQIP, slightly more setup, integrates well with Next.js Image.

Either works. The implementation choice is about toolchain, not about perception.

## When to use

1. **Hero images, photo grids, content cards with imagery.** Anywhere the image is part of the layout the user is reading.
2. **Image galleries.** Each thumbnail benefits; the cumulative latency saved is significant.
3. **User-generated content** where the dimensions are known but the bytes are not yet loaded.

## When *not* to use

1. **Decorative icons or small graphics.** The placeholder overhead is not worth it for a 24×24 icon — just lazy-load it.
2. **Images you don't control the upload pipeline for.** You need to generate the placeholder at upload or build time; if you can't, this pattern is harder to ship cleanly.
3. **Critical above-the-fold imagery on slow networks.** Sometimes the right move is to inline the entire image as base64 — the wait is the bytes, not the placeholder.

## Code

With Next.js Image, the framework can generate and embed a blurred placeholder automatically:

```tsx
import Image from "next/image";

export function Hero({ src, alt }: { src: string; alt: string }) {
  return (
    <Image
      src={src}
      alt={alt}
      width={1280}
      height={720}
      placeholder="blur"
      // For dynamic sources, supply a precomputed blurDataURL:
      // blurDataURL="data:image/jpeg;base64,..."
      sizes="(min-width: 1024px) 1024px, 100vw"
      className="h-auto w-full"
    />
  );
}
```

For a framework-agnostic LQIP, the manual approach:

```tsx
"use client";

import { useState } from "react";

export function ProgressiveImage({
  src,
  lqipSrc,
  alt,
  className,
}: {
  src: string;
  lqipSrc: string;
  alt: string;
  className?: string;
}) {
  const [loaded, setLoaded] = useState(false);

  return (
    <div className={`relative ${className ?? ""}`}>
      {/* Low-quality placeholder, scaled up. Edge contrast preserved
          via slight saturation; the blur smooths over compression
          artifacts without flattening the contrast. */}
      <img
        src={lqipSrc}
        alt=""
        aria-hidden="true"
        className={`
          absolute inset-0 h-full w-full scale-105 object-cover
          blur-md saturate-110 transition-opacity duration-300
          ${loaded ? "opacity-0" : "opacity-100"}
          motion-reduce:transition-none
        `}
      />
      <img
        src={src}
        alt={alt}
        loading="lazy"
        decoding="async"
        onLoad={() => setLoaded(true)}
        className={`
          relative h-full w-full object-cover transition-opacity duration-300
          ${loaded ? "opacity-100" : "opacity-0"}
          motion-reduce:transition-none
        `}
      />
    </div>
  );
}
```

Two details worth noting. First, the `scale-105` on the LQIP hides the soft edges of the blur — without it the boundary is visible. Second, `saturate-110` slightly boosts colour intensity, which compensates for the contrast reduction the blur introduces. Both adjustments are small; together they make the LQIP read as a real preview of the final image.

## Accessibility

1. **`alt=""` and `aria-hidden="true"`** on the placeholder. The real image carries the meaningful alt text; assistive tech should ignore the placeholder.
2. **`loading="lazy"` and `decoding="async"`** on the real image — keep the main thread free during the load.
3. **`prefers-reduced-motion: reduce`** removes the cross-fade transition. The placeholder still appears and disappears, but instantly.
4. **Layout stability** — both elements share the container's width and height so there is no shift when the swap happens. CLS stays at 0.
