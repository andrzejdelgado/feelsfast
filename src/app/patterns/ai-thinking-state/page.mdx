An AI surface that submits a prompt and goes blank for two seconds is broken. The user has no signal that anything is happening; they wonder if the request registered, suspect the system has hung, and start considering a re-submit. The thinking state is the affordance that lives in that gap — between submit and the first token of the response.

The pattern is a styled spinner with semantics. Three pulsing dots, a slow brand-colored shimmer, an avatar with a soft glow — visually it can be many things, but functionally it must communicate *the system is in active process toward your answer*, not *the system is waiting on something it cannot tell you about*. Myers 1985's <Cite id="myers-1985" label="Myers 1985" /> indeterminate-feedback principle is the underlying justification: any visible signal beats none. Fitch's <Cite id="fitch" label="Fitch" /> 1–2 second window is the relevant time band — most AI time-to-first-token sits there, exactly where indeterminate feedback earns its keep.

The thinking state should be replaced as soon as content begins streaming — see [token-by-token streaming](/patterns/ai-token-streaming). It is not an alternative to streaming; it is the bridge from "submitted" to "streaming." Past 5 seconds without first token, escalate to [tool-call transparency](/patterns/ai-tool-call-transparency) or to [engaging loading](/patterns/engaging-loading) — the thinking state alone does not carry that long.

## When to use

1. **Every AI surface with non-trivial time-to-first-token.** Chat composers, inline completions, agent surfaces.
2. **Pair with [token-by-token streaming](/patterns/ai-token-streaming)** — thinking state covers the gap before the first token; streaming covers everything after.

## When *not* to use

1. **Below ~500 ms time-to-first-token.** The thinking state appears, dismisses, and content arrives — three visual events for what would have been one. Tip-the-hand.
2. **Past 5 s without first token.** Escalate to tool-call transparency or engaging loading; the dots alone read as "stalled" past that point.
3. **In a context where the rest of the UI is not AI-related.** A chat-style "thinking" indicator next to a non-AI loader confuses users about what is happening.

## Code

```tsx
"use client";

import { useEffect, useState } from "react";

export function ThinkingDots({ label = "Thinking" }: { label?: string }) {
  return (
    <span
      role="status"
      aria-live="polite"
      aria-label={label}
      className="inline-flex items-center gap-1 text-muted-foreground"
    >
      <Dot delay={0} />
      <Dot delay={200} />
      <Dot delay={400} />
    </span>
  );
}

function Dot({ delay }: { delay: number }) {
  return (
    <span
      aria-hidden="true"
      className="size-1.5 rounded-full bg-muted-foreground motion-safe:animate-pulse"
      style={{ animationDelay: `${delay}ms` }}
    />
  );
}
```

For a pulse-avatar variant — cleaner on chat surfaces where the assistant has its own identity:

```tsx
export function ThinkingAvatar({ src, name }: { src: string; name: string }) {
  return (
    <div className="flex items-center gap-3" aria-live="polite">
      <div className="relative size-8">
        <img
          src={src}
          alt={name}
          className="size-8 rounded-full"
        />
        <span
          aria-hidden="true"
          className="absolute inset-0 rounded-full ring-2 ring-primary/40 motion-safe:animate-ping"
        />
      </div>
      <span className="font-mono text-xs uppercase tracking-wider text-muted-foreground">
        Thinking…
      </span>
    </div>
  );
}
```

The `motion-safe:` gate is mandatory — both variants reduce to a static (but visible) state for `prefers-reduced-motion` users.

## Accessibility

1. **`role="status"` plus `aria-live="polite"`** so screen readers announce the thinking state without interrupting other speech.
2. **`aria-label`** describing what the system is thinking *about* — "Thinking about your question" beats a generic "Thinking".
3. **`motion-safe:`** on every animation; static dots are still informative, just not animated.
4. **Don't trap focus.** The user should still be able to cancel, edit their input, or navigate away.
