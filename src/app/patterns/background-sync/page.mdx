# Background sync

The fastest wait is the one the user does not see. Background sync moves the operation off the user's perceived path entirely — the action commits to local storage immediately, the network request runs whenever the system can deliver it, the user gets on with their day. Conceptually it is the most aggressive perception-layer pattern in the catalog: it does not shorten the wait, it deletes it.

The mechanism is the Service Worker Background Sync API plus a local mutation queue (IndexedDB or similar). The user's action enqueues a mutation, the UI optimistically renders the new state, the worker drains the queue when the network is available. If the user is offline, the queue holds; when connectivity returns, the worker fires. Browser support is good across Chromium and Safari; Firefox lags slightly.

The literature framing is straightforward. Doherty 1982 <Cite id="doherty-1982" label="Doherty 1982" /> establishes that sub-second response converts to measured productivity gains; background sync collapses the response time to *zero* from the user's perspective. Miller 1968 <Cite id="miller-1968" label="Miller 1968" /> tiers operations by visible duration; background sync moves them out of any visible tier.

The trap is the same trap as [optimistic UI](/patterns/optimistic-ui): the rare-but-real failure case needs an honest path. If the worker exhausts retries, the user must learn that the action did not stick — late, but visibly.

## When to use

1. **Mutations the user can recover from if they fail late** — comments, drafts, likes, settings changes.
2. **Offline-tolerant workflows** — note-taking apps, mobile-first interfaces, anything that needs to survive a flaky connection.
3. **Pair with optimistic UI for the local rendering** — the two patterns are designed for each other.

## When *not* to use

1. **Strict-consistency mutations** — payments, irreversible actions, anything where the failure mode of "user moved on, action did not commit" is unacceptable.
2. **High-volume mutations** — queues that grow unbounded eventually fill IndexedDB; design ageing and dropping policies.
3. **Where the user expects a synchronous confirmation** — explicit confirmations ("Send", "Pay", "Submit") that block on success are clearer than background-sync silence.

## Code

A minimal queue-and-drain skeleton. Production implementations belong in libraries (Workbox, Dexie + service worker, Tanstack Query mutation persistence).

```ts
// src/lib/sync-queue.ts
type Mutation = {
  id: string;
  type: string;
  payload: unknown;
  attempts: number;
};

const QUEUE_KEY = "sync-queue:v1";

export async function enqueue(mutation: Mutation) {
  const queue: Mutation[] = JSON.parse(
    localStorage.getItem(QUEUE_KEY) ?? "[]",
  );
  queue.push(mutation);
  localStorage.setItem(QUEUE_KEY, JSON.stringify(queue));
  // Tell the service worker to attempt to drain.
  if ("serviceWorker" in navigator && "SyncManager" in window) {
    const reg = await navigator.serviceWorker.ready;
    await reg.sync.register("drain-mutation-queue");
  } else {
    // Fallback: drain immediately while online.
    void drain();
  }
}

export async function drain() {
  const queue: Mutation[] = JSON.parse(
    localStorage.getItem(QUEUE_KEY) ?? "[]",
  );
  const still: Mutation[] = [];
  for (const m of queue) {
    try {
      await fetch(`/api/${m.type}`, {
        method: "POST",
        body: JSON.stringify(m.payload),
      });
    } catch {
      m.attempts += 1;
      if (m.attempts < 5) still.push(m);
      // Past 5 attempts → surface to the user.
    }
  }
  localStorage.setItem(QUEUE_KEY, JSON.stringify(still));
}
```

In the service worker:

```js
self.addEventListener("sync", (event) => {
  if (event.tag === "drain-mutation-queue") {
    event.waitUntil(drainFromWorker());
  }
});
```

The user-facing UI uses [optimistic UI](/patterns/optimistic-ui) on top — `enqueue()` runs synchronously, the local state updates, the user moves on. The drain happens when it happens.

## Accessibility

1. **The optimistic state must be honest.** If a sync ultimately fails after the queue exhausts retries, surface it visibly (toast, inline error on the affected item, retry control). Silent failures break trust.
2. **Don't announce queue successes.** A "synced" toast for every enqueued mutation becomes noise. Announce only the failures.
3. **Provide a way to inspect the queue.** A power-user setting that shows pending and failed mutations is good practice for offline-first apps.
4. **`prefers-reduced-data` users may not want background drain on metered connections** — gate aggressive draining accordingly.
