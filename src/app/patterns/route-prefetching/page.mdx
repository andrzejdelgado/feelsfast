# Route prefetching

<RelatedRail of="route-prefetching" kind="pattern" />

A user clicks a link. The browser fetches the route bundle and data, the server responds, the new page renders. With route prefetching, the bundle and data are already warm by the time the click happens — sometimes already cached, sometimes mid-flight. The navigation feels instant or near-instant; the wait the user *would have* experienced never registers.

Most modern frameworks ship this as a default. Next.js, Astro, SvelteKit, and Remix all prefetch linked routes on viewport entry; some also support hover-triggered prefetching. The pattern lands in the catalog because the *triggers* are tunable, and tuning them is one of the cheaper perception-layer wins available.

The triggers, ordered by aggressiveness:

1. **Viewport entry.** As soon as the link scrolls into view, prefetch. Cheapest signal of intent; works for most consumption surfaces.
2. **Hover.** The cursor crossing the link's bounding box triggers the prefetch. Mid-aggressiveness; works on desktop.
3. **Mouse deceleration.** Eli Fitch's <Cite id="fitch" label="Fitch" /> Future Link technique — the cursor's velocity vector is decelerating toward the link, indicating an upcoming click. Most aggressive; gets you ~600 ms of additional head-start in the right conditions.
4. **`pointerdown` / `mousedown`.** The user has committed to clicking, ~100–150 ms before they release. Covered separately in [mousedown / pointerdown vs. click](/patterns/mousedown-vs-click).

Pair them. Viewport-entry on every link, hover on isolated CTAs, mouse-deceleration on primary actions, `pointerdown` everywhere a network request is involved. Each step shrinks the visible wait without making the system itself faster — Miller 1968 <Cite id="miller-1968" label="Miller 1968" /> is the underlying threshold being moved closer to the user.

## When to use

1. **Consumption surfaces** — marketing pages, documentation, dashboards, news feeds. Prefetch is essentially free for these.
2. **Routes whose payloads are stable across users** (anonymous routes, public pages). The prefetched bytes are reusable.
3. **Multi-step flows** (wizards, checkouts) where the next step is predictable from the current one.

## When *not* to use

1. **Routes with side-effects on load.** If hitting the route triggers analytics, server-side caching, or a state mutation, prefetching every visible link will inflate those metrics with phantom traffic.
2. **Authenticated routes whose payloads vary by user** if the user has not yet authenticated. Prefetching them will either fail or return the wrong data.
3. **Slow networks where prefetching competes with the current page's resources.** The user agent's connection-aware heuristics handle most of this, but heavy-handed manual prefetching can still hurt the active page.

## Code

```tsx
// Next.js — prefetch defaults to on for App Router <Link> components in
// the viewport. Disable per-link when you want to avoid the request:
import Link from "next/link";

<Link href="/scenarios/page-load-cold" prefetch={false}>
  Page load (cold)
</Link>
```

For frameworks without built-in prefetch, the manual primitive:

```tsx
"use client";

const prefetched = new Set<string>();

function prefetch(href: string) {
  if (prefetched.has(href)) return;
  prefetched.add(href);

  // For documents: a <link rel="prefetch"> tells the browser to fetch
  // and cache the resource at low priority.
  const link = document.createElement("link");
  link.rel = "prefetch";
  link.href = href;
  link.as = "document";
  document.head.appendChild(link);
}

export function PrefetchOnHover({
  href,
  children,
}: {
  href: string;
  children: React.ReactNode;
}) {
  return (
    <a
      href={href}
      onMouseEnter={() => prefetch(href)}
      onFocus={() => prefetch(href)}
    >
      {children}
    </a>
  );
}
```

Note `onFocus` alongside `onMouseEnter` — keyboard users tab through links, and the focus event is the right hover-equivalent for them.

## Accessibility

1. **Trigger on `onFocus` as well as `onMouseEnter`.** Otherwise keyboard users get none of the benefit.
2. **`prefers-reduced-data`** is an emerging media query for users on metered connections — prefetching aggressively contradicts the user's intent. Where the framework supports it, gate the more aggressive triggers (deceleration, mousedown) on `not (prefers-reduced-data)`.
3. **Screen-reader announcements** are unaffected — the prefetch is invisible. The benefit shows when the user actually navigates: the route is already there.
