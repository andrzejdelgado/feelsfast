A determinate progress bar reports actual progress through a known total — bytes transferred, files processed, items rendered. It is the most-cited "should I show a loader?" pattern in the literature, and the source-of-record is older and more boring than the modern UX folklore suggests. Myers 1985 <Cite id="myers-1985" label="Myers 1985" /> ran a small but careful CHI study and found ~86 % of participants preferred a percent-done bar over a blank wait — and the preference held even when the bar was inaccurate.

Two things follow from that result. First, a determinate bar is the default to reach for when you have real progress data and the wait sits in the 1–10 s band. Second, a *fake* progress bar — one that animates to completion against a fixed CSS duration regardless of actual progress — is worse than no bar at all. It will desync from reality and either rush the user (finishing while loading continues) or lie to them (sitting at 95 % while the wait drags on). Myers's preference holds for indicators that *might* be inaccurate; it does not extend to indicators that are *deliberately* inaccurate.

The animation pattern matters too. Harrison et al. 2010 <Cite id="harrison-2010" label="Harrison et al. 2010" /> measured a ~11–12 % perceived speed-up when the bar's bands move backwards and decelerate over the duration — covered separately in the [animated progress pattern](/patterns/animated-progress).

## When to use

1. **1–10 s waits where progress is measurable.** File uploads, downloads, multi-step processing, batch operations.
2. **The duration can be estimated within reasonable margins.** A wait that might be 2 s or might be 30 s does not benefit from a determinate bar — the user feels the lie.
3. **The user has decided to wait and wants to know how much more.** This is the cognitive condition Myers's study was set up around.

## When *not* to use

1. **Sub-1-second waits.** Tip-the-hand. The bar appears, dismisses, content arrives — three visual events for what would have been one.
2. **Operations with genuinely unknown duration.** Use an indeterminate spinner (in the 1–2 s window) or a skeleton screen (1–10 s with stable layout).
3. **Fake progress with a fixed CSS duration.** This is the failure mode Myers's result does not protect. Either reflect reality or do not show a bar.

## Code

```tsx
type ProgressProps = {
  /** 0–1; clamp inside the component. */
  value: number;
  /** Optional human-readable label for screen readers. */
  label?: string;
};

export function ProgressBar({ value, label }: ProgressProps) {
  const clamped = Math.max(0, Math.min(1, value));
  const percent = Math.round(clamped * 100);

  return (
    <div
      role="progressbar"
      aria-label={label ?? "Progress"}
      aria-valuemin={0}
      aria-valuemax={100}
      aria-valuenow={percent}
      className="h-2 w-full overflow-hidden rounded-full bg-muted"
    >
      <div
        className="h-full bg-primary transition-[width] duration-200 ease-out"
        style={{ width: `${percent}%` }}
      />
    </div>
  );
}
```

The ARIA contract here is the part most teams skip. Without `role="progressbar"` plus the `aria-value*` attributes, screen reader users get nothing — they hear "loading" once and then silence. With them, assistive technology can announce progress changes.

For long operations, also announce milestones in a polite live region:

```tsx
<div aria-live="polite" className="sr-only">
  {percent === 100 ? "Upload complete" : null}
  {percent === 50 && previousMilestone < 50 ? "Halfway done" : null}
</div>
```

## Accessibility

1. **`role="progressbar"`** on the container plus `aria-valuemin / aria-valuemax / aria-valuenow`. The trio is what assistive tech reads.
2. **`aria-label`** describing what is loading. "Progress" alone is not enough; "Uploading file" is.
3. **Motion-respecting transition.** The width transition is a CSS property change, not an animation — `prefers-reduced-motion` reduces it automatically. If you add a band animation (per Harrison et al.), gate it with `motion-reduce:animate-none`.
4. **Polite live region** for milestone announcements (start, halfway, complete) so screen-reader users get the same affordances sighted users do.
5. **Visible focus state** on any cancellation button you ship alongside the bar.
