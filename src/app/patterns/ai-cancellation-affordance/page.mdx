# Cancellation affordance

The user submits a prompt, watches the assistant generate, and realises the response is going somewhere they didn't want. The stop button is what saves the interaction. Done well, it makes long AI operations feel safe — the user can commit to running anything because they can always pull out. Done poorly, it confirms the user's worst suspicion about AI: that they are passengers, not drivers.

Doherty 1982's <Cite id="doherty-1982" label="Doherty 1982" /> productivity-cliff finding has a less-cited corollary: productivity rises when the user feels in control of response time. The cancel button is the strongest single signal of control on an AI surface. Card-Moran-Newell 1983's <Cite id="card-moran-newell-1983" label="Card, Moran & Newell 1983" /> ~100 ms perceptual frame sets the latency ceiling — if pressing stop takes longer than ~100 ms to register, the cancellation itself feels broken.

Two contracts the affordance needs to honour. First, the stop must register within the perceptual frame — visible state change in under 100 ms, even if the actual abort takes longer to propagate. Second, mid-stream cancellation must not lose what the user has so far. A user who cancels mid-token-stream still wants to keep the partial response, edit their prompt, or save what came back. Silent loss-of-state is worse than a slow cancel.

## When to use

1. **Every long-running AI operation** — chat streaming, agentic workflows, long inferences, batch generation.
2. **Anywhere the operation could go off-track** in ways the user might want to abort early.
3. **Pair with [tool-call transparency](/patterns/ai-tool-call-transparency)** — visible work plus visible cancellation closes the loop on user agency.

## When *not* to use

1. **Operations under ~1 second** where the cancel UI flickers in and out faster than the user can read it.
2. **Truly atomic operations** that cannot be safely interrupted (a payment, an irreversible commit). Even there, surface the irreversibility *before* the operation starts.

## Code

```tsx
"use client";

import { useEffect, useRef, useState } from "react";

export function CancellableStream({
  prompt,
}: {
  prompt: string;
}) {
  const [text, setText] = useState("");
  const [streaming, setStreaming] = useState(true);
  const [cancelled, setCancelled] = useState(false);
  const controllerRef = useRef<AbortController | null>(null);

  useEffect(() => {
    const controller = new AbortController();
    controllerRef.current = controller;

    async function run() {
      try {
        const response = await fetch("/api/chat", {
          method: "POST",
          body: JSON.stringify({ prompt }),
          signal: controller.signal,
        });
        const reader = response.body?.getReader();
        if (!reader) return;
        const decoder = new TextDecoder();
        let acc = "";
        while (true) {
          const { value, done } = await reader.read();
          if (done) break;
          acc += decoder.decode(value);
          setText(acc);
        }
      } catch (error) {
        if ((error as Error).name === "AbortError") {
          // Expected; the user pressed stop.
        } else {
          throw error;
        }
      } finally {
        setStreaming(false);
      }
    }

    void run();
    return () => controller.abort();
  }, [prompt]);

  const stop = () => {
    // Visible state flips immediately — the perceptual contract.
    setCancelled(true);
    setStreaming(false);
    // Then signal the underlying request to abort.
    controllerRef.current?.abort();
  };

  return (
    <div className="space-y-3">
      <div aria-live="polite" aria-busy={streaming}>
        {text}
        {cancelled ? (
          <span className="ml-1 font-mono text-xs uppercase tracking-wider text-muted-foreground">
            stopped
          </span>
        ) : null}
      </div>
      {streaming ? (
        <button
          type="button"
          onClick={stop}
          className="rounded-md border border-border bg-background px-3 py-1.5 text-xs font-medium hover:bg-secondary"
        >
          Stop generating
        </button>
      ) : null}
    </div>
  );
}
```

The order of operations in `stop` matters. The visible flip (`setCancelled`, `setStreaming`) happens *first*, synchronously inside the click handler. The actual `controller.abort()` runs after — the network teardown can take longer than 100 ms, but the user already has their acknowledgment.

For agentic workflows where cancellation has implications (a half-applied file edit, a partially-completed multi-tool plan), the cancel handler should also surface what was preserved and what was rolled back.

## Accessibility

1. **The stop button needs sufficient contrast and hit-target size** — direct-manipulation precision applies to dismissals as much as to ignitions. ≥44 × 44 px touch target.
2. **Keyboard shortcut** — Escape should cancel by convention; surface it visibly the first few times the user uses the affordance.
3. **`aria-live="polite"`** on the container so screen readers announce the cancelled state.
4. **The cancelled state should be visually distinct from the in-progress and completed states.** A grayed "stopped" label, a different-coloured indicator, or a position change on the message — but never silent.
