# mousedown / pointerdown vs. click

When a user clicks a button, their finger or mouse button does not snap up and down instantly. They press, hold for ~100–150 ms, and release. The `click` event fires on release; `mousedown` and `pointerdown` fire on press. If your handler sits on `click`, you have spent that 100–150 ms doing nothing while the user was already committed to the action.

Eli Fitch <Cite id="fitch" label="Fitch" /> measured the gap directly. His Mechanical Turk study had users clicking buttons in a controlled UI and recorded median hold times in the 100–150 ms range — so listening on `mousedown` instead of `click` is essentially free latency budget. He also noticed an incidental ~50 ms behavioural extension when the button has visible press feedback ([pre-action feedback](/patterns/pre-action-feedback) covers this) — bringing the total head-start to ~150–200 ms in practice.

The pattern is most valuable when the work the handler kicks off has measurable cost — fetching a route, prefetching a bundle, beginning a network request. For pure UI-state toggles (a checkbox, a tab switch) the gain is invisible; for anything that touches the network, it lands inside Doherty's productivity-cliff territory.

## When to use

1. **Anywhere `click` was going to fire a network request.** Route navigation, form-step preloading, search submission, AI prompt submission — every one of these can begin on press.
2. **Buttons with visible press feedback.** The combination of feedback + early start gives you the full ~200 ms head-start Fitch documented.
3. **Idempotent operations.** If the user slides off the button mid-press, the operation should be safe to abandon. Loading a JavaScript bundle is fine; charging a credit card is not.

## When *not* to use

1. **Side-effects with cost or commitment.** If the user starts to press and slides off, you should not have shipped a payment, sent a message, or deleted a file. Reserve those for `click` or for an explicit confirmation.
2. **Drag-and-drop targets.** `mousedown` triggers the drag; firing the click action on `mousedown` would short-circuit the drag entirely.
3. **Touch interactions without a `touchmove` guard** — see below.

## Code

```tsx
"use client";

import { useRef } from "react";

export function PrefetchOnPress({
  href,
  children,
}: {
  href: string;
  children: React.ReactNode;
}) {
  return (
    <a
      href={href}
      onPointerDown={() => {
        // Begin prefetching the route immediately. The user has not yet
        // released — but they have committed to the press.
        void preloadRoute(href);
      }}
    >
      {children}
    </a>
  );
}
```

For touch interactions, the press-on-touchstart pattern needs a `touchmove` gate. Otherwise scrolling triggers your action whenever a finger lands on a button on its way past:

```tsx
"use client";

import { useRef, type TouchEvent } from "react";

const TOUCH_MOVE_THRESHOLD_PX = 6;

export function TouchSafePress({
  href,
  children,
}: {
  href: string;
  children: React.ReactNode;
}) {
  const startedAt = useRef<{ x: number; y: number } | null>(null);
  const cancelled = useRef(false);

  const onTouchStart = (event: TouchEvent<HTMLAnchorElement>) => {
    const touch = event.touches[0];
    startedAt.current = { x: touch.clientX, y: touch.clientY };
    cancelled.current = false;
    // Begin the work — same head-start as the mouse case.
    void preloadRoute(href);
  };

  const onTouchMove = (event: TouchEvent<HTMLAnchorElement>) => {
    if (!startedAt.current || cancelled.current) return;
    const touch = event.touches[0];
    const dx = touch.clientX - startedAt.current.x;
    const dy = touch.clientY - startedAt.current.y;
    if (Math.hypot(dx, dy) > TOUCH_MOVE_THRESHOLD_PX) {
      // The user is scrolling, not tapping — abandon the prefetch.
      cancelled.current = true;
    }
  };

  // The actual navigation runs on the standard `click` (which fires only on
  // a touchend that didn't cross the move threshold). The prefetched bundle
  // is already warm; the navigation is now near-instant.
  return (
    <a href={href} onTouchStart={onTouchStart} onTouchMove={onTouchMove}>
      {children}
    </a>
  );
}
```

The threshold matters. Direct-touch input has tight latency JNDs — Jota et al. <Cite id="jota-2013" label="Jota et al. 2013" /> measured ~33 ms for dragging and ~82 ms for tapping, and Deber et al. <Cite id="deber-2015" label="Deber et al. 2015" /> confirmed and extended those numbers — so the move-threshold guard needs to be cheap. The 6-pixel threshold above is conservative; for some surfaces 8–10 pixels feels right.

## Accessibility

1. **Keyboard `Enter` and `Space` should still trigger the action** the same way `click` would. The pattern is for pointer / touch input; keyboard fires on key release by convention and that is fine.
2. **Don't break long-press affordances.** If your control supports right-click or long-press for a context menu, the `mousedown` head-start should not interfere.
3. **Honour `prefers-reduced-motion`** in any press animation that fires alongside the prefetch — covered in [pre-action feedback](/patterns/pre-action-feedback).
4. **Don't fire screen-reader announcements on mousedown.** The user has not yet committed; an announcement before they release is premature.
