# Stale-while-revalidate

<RelatedRail of="stale-while-revalidate" kind="pattern" />

A user requests a resource. The cache has it, but the cached copy is past its freshness window. The classical response is to invalidate, fetch, and serve the fresh response — incurring the full network round-trip on the user's perceived load time. Stale-while-revalidate (SWR) flips the priority: serve the stale copy immediately, fetch the fresh copy in the background, swap it in when ready.

The user sees instant feedback in the perceptual frame Card-Moran-Newell describe; the system catches up to current data inside Miller's <Cite id="miller-1968" label="Miller 1968" /> simple-inquiry tier. On cached responses, the entire perceived wait collapses into the cache-hit rendering. Doherty's <Cite id="doherty-1982" label="Doherty 1982" /> productivity-cliff finding is what makes this such a high-leverage pattern: every cache hit that crosses the sub-second boundary is essentially free productivity.

The pattern is now a first-class HTTP cache directive (`Cache-Control: stale-while-revalidate=…`), a standard React data-fetching mode (TanStack Query, the SWR library, RTK Query), and a default in most edge-rendering platforms. The implementation is mature; the perception case is what designers should know.

## When to use

1. **Read-mostly data with tolerable staleness windows** — pricing pages, feeds, dashboards where last-minute freshness is preferred but not critical.
2. **Per-user data with consistent shape** — user profiles, settings, preferences.
3. **AI / inference responses with caching** — the same prompt with stable parameters can serve cached output while a fresh inference runs in the background.

## When *not* to use

1. **Strict-consistency reads.** Banking balances, inventory counts before purchase, real-time game state — anywhere stale data could mislead the user into a decision they would not otherwise make.
2. **Initial loads with no cached value to fall back to.** SWR is a cache-hit pattern; the first request always pays the full latency.
3. **Highly volatile data with very short freshness windows.** If staleness is measured in seconds, the SWR window collapses; you are effectively making two requests for every one.

## Code

```tsx
// React + TanStack Query — SWR is the default behaviour.
"use client";

import { useQuery } from "@tanstack/react-query";

export function Profile({ userId }: { userId: string }) {
  const { data, isLoading, isFetching } = useQuery({
    queryKey: ["profile", userId],
    queryFn: () => fetchProfile(userId),
    // Serve cached copies up to 5 minutes old immediately.
    staleTime: 5 * 60 * 1000,
    // Keep them around 30 minutes after they fall out of staleTime,
    // so navigations that bring the user back still get instant render.
    gcTime: 30 * 60 * 1000,
  });

  if (isLoading) {
    // No cache to serve — first load. Skeleton screen lands here.
    return <ProfileSkeleton />;
  }

  return (
    <div className={isFetching ? "opacity-60 transition-opacity" : ""}>
      <h2>{data!.name}</h2>
      <p className="text-muted-foreground">{data!.bio}</p>
    </div>
  );
}
```

The `isFetching` flag is the SWR signal — the cache served, but a background revalidation is in flight. The dim-stale-content pattern makes that visible without blocking the user.

For HTTP-level SWR, the cache header tells the platform layer to do the same:

```http
Cache-Control: max-age=60, stale-while-revalidate=600
```

The CDN serves cached responses up to 60 seconds old fresh, up to 660 seconds old stale (with background revalidation), and only fetches synchronously past that.

## Accessibility

1. **Don't announce staleness changes** in a live region. They are noise the user does not need.
2. **Do announce when the resolved fresh data is *substantially* different** — a price change, a status change, anything the user would want to act on. Use `role="status"` for non-critical updates, `role="alert"` for critical ones.
3. **The stale-content dim should respect `prefers-reduced-motion`** — keep the opacity change but consider removing the transition.
4. **Match the cache lifetime to user expectations.** If the user has manually refreshed, treat that as a hard cache bust — they wanted fresh data, not your SWR window.
