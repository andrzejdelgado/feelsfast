# Empty states that hint at loading

A common failure mode in loading affordance: the page renders, shows nothing, and the user cannot tell whether the load is still happening or whether the load finished and there are no items. Both states are blank. Both states look identical. The user waits, then guesses.

The fix is small but important. Distinguish loading-empty from settled-empty. Myers 1985 <Cite id="myers-1985" label="Myers 1985" /> framed the broader principle: any feedback beats none. The empty-state-loading pattern is the corollary applied to the specific case where the *resolved* state happens to also be empty.

Two related cases. First, the content area resolves with zero items — show a settled empty state (illustration, helpful text, suggested action). Second, the content area is still loading — show a loading affordance ([skeleton](/patterns/skeleton-screens), [spinner](/patterns/spinners), or [progress](/patterns/determinate-progress) per the time band). The two should look obviously different.

This sounds obvious. It is not. Most teams skip it because the loading state and the settled empty state get hooked up at different times, often by different developers, and nobody steps back to compare them.

## When to use

Always, on any content surface that can be empty after a successful load.

## When *not* to use

There is essentially no when-not-to-use. The closest exception is when the content is *guaranteed* to have items — a `popular items` carousel where business logic ensures non-empty results. Even there, plan for the failure case: API errors that surface as empty.

## Code

```tsx
type ContentAreaProps<T> = {
  items: T[] | undefined;
  isLoading: boolean;
  error?: Error;
  renderItem: (item: T) => React.ReactNode;
  emptyState: React.ReactNode;
};

export function ContentArea<T>({
  items,
  isLoading,
  error,
  renderItem,
  emptyState,
}: ContentAreaProps<T>) {
  if (isLoading) {
    return <ListSkeleton count={5} />;
  }
  if (error) {
    return <ErrorState error={error} />;
  }
  if (!items || items.length === 0) {
    return <>{emptyState}</>;
  }
  return <ul>{items.map((item) => renderItem(item))}</ul>;
}
```

The `EmptyState` component itself should communicate three things: that the load completed successfully, that the area is empty, and what the user can do about it.

```tsx
export function EmptyState() {
  return (
    <div className="flex flex-col items-center gap-3 py-12 text-center">
      <Inbox aria-hidden="true" className="size-10 text-muted-foreground" />
      <p className="text-sm font-medium">No items yet</p>
      <p className="max-w-sm text-sm text-muted-foreground">
        Items you add will show up here. Try adding your first one.
      </p>
      <button
        type="button"
        className="mt-2 rounded-md bg-primary px-3 py-1.5 text-sm font-medium text-primary-foreground"
      >
        Add an item
      </button>
    </div>
  );
}
```

The icon, the heading, the description, and the call to action together signal "the system is in a settled state, here is what to do." The skeleton next to it signals "the system is still working, please wait." The two are visually distinct, which is the whole point.

## Accessibility

1. **`aria-busy="true"`** on the container during the loading state; flip to `false` once content arrives (whether items or empty state).
2. **`role="status"`** on the empty-state container so screen readers announce it as a transient status. Or keep it semantic if it is the entire page main, then no extra role.
3. **Focus management on transition**: if the user triggered the load, move focus to the resolved content (or to the empty state's CTA) so keyboard users land where the action is.
4. **Don't use the same icon or copy** for empty and error states. Errors are recoverable; empty states are baseline.
